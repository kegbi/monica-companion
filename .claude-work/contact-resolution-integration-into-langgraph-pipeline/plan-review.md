### MEDIUM

1. [MEDIUM] Missing OTel span instrumentation in the new node -- All existing graph nodes wrap their logic in `tracer.startActiveSpan(...)` and call `span.end()` in a `finally` block. The `node-spans.test.ts` file tests this pattern for every node. The plan's Step 3 does not mention adding an OTel span to `resolveContactRef`, and Step 2 does not include `node-spans.test.ts` in the list of test files that need a new span test for the new node. -- **Fix:** Add to Step 3: wrap the node body in `tracer.startActiveSpan("ai-router.graph.resolve_contact_ref", ...)` with `span.end()` in a `finally` block. Record span attributes for `ai-router.resolution_outcome` (resolved/ambiguous/no_match/skipped). Add a corresponding span test in `node-spans.test.ts`.

2. [MEDIUM] Graceful degradation maps fetch failure to `no_match`, losing the error signal -- Step 3 states: "On `fetchContactSummaries` failure: log warning, set `contactResolution` to a no_match result." This makes the user unable to distinguish between "genuinely no matching contacts" and "the monica-integration service was unreachable." -- **Fix:** When `fetchContactSummaries` fails, skip contact resolution entirely and return `{}` (no state changes). This lets the LLM's original `contactRef` and `commandPayload` pass through unmodified to `executeAction`, preserving the pre-integration behavior as a true graceful degradation. Log the failure for observability.

3. [MEDIUM] `userFacingText` for `no_match` outcome is generated by the node, not the LLM -- Step 3 says that on `no_match`, the node should "update `userFacingText` to ask for clarification." The LLM already produced a `userFacingText` in the classification step, in the user's detected language. Overriding it from the deterministic node means the user gets a hardcoded English prompt instead of a language-appropriate LLM-generated response. -- **Fix:** For `no_match`, set `needsClarification = true` and `clarificationReason = "ambiguous_contact"` but preserve the LLM's original `userFacingText`.

### LOW

1. [LOW] Plan language says "mutate intentClassification" but LangGraph returns state updates -- The node should construct a modified copy of `intentClassification` and return it as part of the update object, using spread syntax.

2. [LOW] `contactSummariesCache` adds state bulk for a field that cannot be hit in the current linear graph -- Consider deferring until graph topology changes require it. Advisory only.

3. [LOW] Redundant `create_contact` skip check -- The `contactRef === null` check already covers `create_contact` commands. The explicit check is a defensive belt-and-suspenders guard.

4. [LOW] Fabricated `contactId` values from the LLM pass through unchecked when resolution is skipped -- Pre-existing issue, defer to a separate hardening task.
